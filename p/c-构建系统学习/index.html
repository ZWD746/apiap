把简单的问题做复杂

引言：

在我读高中时期，我的物理老师曾经说过一句话"把复杂的问题弄简单的是高手，把简单问题弄复杂的也是高手”。

曾经我对这句话的前半段无比认可，当时的我认为如果一个人能将一个复杂的问题简单化，那他一定十分绝对是这个领域的强者，并且愿意花时间把这个领域的知识通俗化，向还未入门的人来传播。当时的我认为这样的人不仅实力强悍，内心也很强大，而把简单问题复杂化的人，都是故弄玄虚，人为设置门槛，想让自己在别人眼里“看起来"很厉害，说到底是内心不自信的表现。

在现在我的看来，这后半句话也是有些道理的，在某些工作中，是要将一些简单的问题复杂化，但是这个复杂是针对于局部的，通过让一些局部问题复杂化，来使得这个问题的整体变得清晰化，统一化。这样对于问题的解决就有了可以规范化的道路，让整个问题更容易去解决。虽然这会提升解决这个问题的门槛，但是对于已经入门的人来说，这反而增加了解决问题的效率，正如我接下来要学习的构建系统。

第一个使用构建系统的cpp工程——hello world

这次我依然使用行业惯例——Hello world，作为我的第一个有构建系统的工程。

以下就是一个简单的hello world项目的目录，和代码

#include <iostream>

int main()
{
    std::cout << "Hello world" << std::endl;

    return 0;
}

相信大家再熟悉不过了，但是在实际工作中，这种项目几乎不存在。把所有的功能模块放到一个文件里，这既不利于开发，也不利于维护。总不能在开发的时候把一个文件传来传去，你写一会你的代码，我写一会我的代码。效率极其低下，所以我们要解耦合。

此时我们将目录改为如下所示

#include "hello/hello.h"
#include "world/world.h"

int main()
{
    printHello();
    printWorld();

    return 0;
}

此时main函数内容如上所示，将打印hello和打印world分别解耦到两个不同的模块，这样这三个模块就可以同时开发了。于此同时，编译的难度又上升了。

对于原先的项目结构，我们只需输入一下编译命令即可

1

g++ -o helloWorld main.cpp

而现在我们需要输入

1

g++ -o helloWorld main.cpp hello/hello.cpp world/world.cpp

对于我们这个小项目，这么做只是稍微有那么一点点麻烦，凡是，对于大型项目来说，这么做无疑是灾难级的。一个几十、上百个模块的项目，且不论我们写编译命令时要写多少个.cpp文件，编译成功后，若改动其中一个小模块，则需要整个项目重写编译，这造成了大量的时间浪费。

那么，有没有办法避免呢？当然时有的，我们可以分别编译每一个模块，最后再链接到一起。

1
2
3
4

g++ -c main.cpp
g++ -c hello/hello.cpp
g++ -c world/world.cpp
g++ -o helloWorld main.o hello.o world.o

这样我们在修改了某个模块后，只需要重写编译该模块的.o文件，然后在重写链接即可。这样虽然解决了“牵一发而动全身”的问题，但是对于编译来说，仍有不少麻烦。我们人要一个一个手动编译模块，这就会牵扯到遗漏的情况。于是我们引入makefile来解决问题。

我们需要在项目目录下创建一个Makefile文件(文件名及大小写必须和我写的一致)，内容填写

        g++ -o helloWorld main.o hello.o world.o

main.o: main.cpp hello/hello.h world/world.h
        g++ -c main.cpp

hello.o: hello/hello.h hello/hello.cpp
        g++ -c hello/hello.cpp

world.o: world/world.h world/world.cpp
        g++ -c world/world.cpp

项目目录变为

此时在Makefile同目录下，只需输入一个make，系统便根据所写Makefie文件来执行编译命令，这个编译时增量编译，当你修改你个文件时，只有依赖他的模块会重新编译，其余则不会变化，这大大减少了浪费的时间，同时也将手动逐个编译，变成自动编译。
